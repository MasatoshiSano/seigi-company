# チームリーダーのための Claude Code チーム運用ガイド

秘書室・技術ナレッジ部署が参照するナレッジ。
リーダーが開発チームで Claude Code を導入・運用する際の実践的な手順と判断基準。

---

## 1. リーダーがまずやること（初日）

### CLAUDE.md（チームの教科書）を作る

プロジェクトのルートに `CLAUDE.md` を置く。
AIに対する「うちのチームではこうやってね」という指示書。

```markdown
# CLAUDE.md

## プロジェクト概要
- 何を作るか（1〜2行で）
- 技術スタック

## 開発コマンド
- `npm run dev` — 開発サーバー起動
- `npm run test` — テスト実行

## コーディングルール
- （チームで守るべきルール）

## やってはいけないこと
- `.env` をコミットしない
- `main` に直接プッシュしない
```

**最初は15〜30行で十分。** 開発しながらチーム全員で育てる。

### ガードレール（安全装置）を設定する

`.claude/settings.json` を作成して Git で共有。

```json
{
  "permissions": {
    "deny": ["rm -rf", "git push --force", "DROP TABLE"],
    "ask": ["git push", "git merge"]
  }
}
```

- `deny` → 絶対に実行させない操作
- `ask` → 毎回人間の確認を求める操作

## 2. チームメンバーの経験値に応じた設定

各メンバーは `CLAUDE.local.md`（個人用、Git共有されない）を作る。

| メンバーの経験 | CLAUDE.local.md に書く内容 |
|-------------|-------------------------|
| 新人 | 「コメントで理由を説明して」「知らないライブラリは解説して」 |
| 中堅 | 「設計判断は選択肢を出して」「パフォーマンスに注意して」 |
| ベテラン | 「説明は最小限。コードで示して」 |

**チーム共通のルールはCLAUDE.md、個人の好みはCLAUDE.local.md。**

---

## 3. CLAUDE.md の育て方

### 追記のタイミング

- AIが間違えたとき（同じミスを二度起こさせない）
- チーム内でフォーマットがバラバラになったとき
- 「これ毎回説明してるな」と気づいたとき

### 追記の具体例

```
問題：日付の表示形式がメンバーごとにバラバラ
追記：「画面表示は YYYY年MM月DD日、APIは ISO 8601」

問題：APIエラーの返し方が統一されていない
追記：「エラーは { error: { code, message } } 形式で統一」

問題：新人がテストなしでコードを書いた
追記：「テストなしのコードはマージしない」
```

### 運用ルール

- CLAUDE.md の変更は**PRレビューを通す**（チーム全員に影響するため）
- **200行以下**に保つ。超えそうなら詳細は別ファイルに分離
- 毎週のふりかえりで「今週Claudeが間違えたこと」を確認して追記

## 4. スキル（手順書）の活用

繰り返す作業は `.claude/skills/` にスキルファイルとして保存。
誰がやっても同じ品質で作業できるようになる。

### リーダーが用意すべきスキルの例

| スキル名 | 用途 |
|---------|------|
| `/create-page` | 新しい画面を作るときの手順 |
| `/create-api` | 新しいAPIを作るときの手順 |
| `/review` | コードレビューのチェックリスト |
| `/create-pr` | PR作成の手順と記述テンプレート |

---

## 5. 並列開発（リーダーのスピードアップ技）

### Git Worktree で複数AIを同時に走らせる

```bash
claude --worktree  # 独立した作業場所でAIを起動
```

- 1人で4〜5個のAIを並列実行可能
- 各AIは別の作業場所で動くので互いに干渉しない
- リーダーは各成果物を確認して取り込むだけ

### 判断基準

| 状況 | 推奨 |
|------|------|
| 締め切りが近い | 並列開発で機能を同時進行 |
| 独立した機能が複数ある | 機能ごとにWorktreeを作成 |
| 機能同士が密結合 | 並列化せず順番に開発 |

---

## 6. 毎週のふりかえりテンプレート

```markdown
## 週次ふりかえり（Claude Code運用）

### 今週Claudeが間違えたこと
- （メンバーから収集）

### CLAUDE.mdに追記すべきルール
- （上記から抽出）

### スキル化すべき繰り返し作業
- （チームで議論）

### 来週の並列化候補
- （独立した作業のリストアップ）
```

## 7. 企業事例（参考）

| 企業 | 重視したこと | 結果 |
|------|------------|------|
| Plaid（5名チーム） | 手順の標準化 | 月のPR数が4倍 |
| incident.io | 並列開発のスピード | 2時間→10分の事例 |
| Every | 学習ループの蓄積 | 使うほどAIが賢くなる仕組み |

**どの事例にも共通するのは「CLAUDE.mdをGit管理してチーム全員で育てる」こと。**
