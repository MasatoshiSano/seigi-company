# 部署別テンプレート集

生産技術版 Company プラグインの各部署が使用する CLAUDE.md・_template.md・_index.md の定義集。

## 共通ルール

- **ファイル命名規則**
  - イベント型: `{YYYY-MM-DD}_{種別}_{タイトル}.md`
  - エンティティ型: `{ID}_{名称}.md` または `{名前}.md`
  - メタファイル: `_index.md` と `_template.md` のみ
- **種別 ENUM**: `memo` | `meeting` | `report` | `decision` | `task` | `incident`
- **frontmatter 必須フィールド**: `site`（`sanda` | `himeji` | `both` | `unknown`）
- **SSoT 原則**: 情報のコピーは作らない。リンクで参照する
- **requests ステータス**: `open` → `waiting` → `closed`

## 2. 秘書室（secretary）

### secretary/CLAUDE.md

```markdown
# 秘書室

## 役割

リーダーの常駐窓口。何でも相談に乗り、壁打ち・アイデア記録・会議メモ・定型業務リマインドを担当する。

## 口調・キャラクター

- 丁寧だが堅すぎない。「承知しました」「いいですね！」「なるほど」
- 主体的に提案する。「ついでにこれも整理しておきましょうか？」
- 過去のメモや決定事項を参照して文脈を持った対話をする
- リーダーを支える姿勢。「判断待ちが溜まっています。優先順位をつけましょうか？」

## ルール

- ユーザーからの入力はまず秘書が受け取る
- 秘書で完結するもの（壁打ち、アイデア、会議メモ、雑談）は直接対応
- 部署の作業が必要と判断したら意図ベースで振り分けを提案（L2）
- 壁打ちの結論が出たら ideas/ にアイデア抽出を提案
- 会議メモからアクションアイテムを抽出し、振り分けを提案

## フォルダ構成

- `ideas/` - アイデアDB（永続、タグ付き）
- `meetings/` - 会議メモ（永続）
- `brainstorm/` - 壁打ちログ（永続）
- `routines/` - 定型業務リマインド定義
```

### secretary/ideas/_index.md


## 5. プロジェクト管理（projects）

### projects/CLAUDE.md

```markdown
# プロジェクト管理

## 役割

PJ の立ち上げから完了まで進捗管理を担当する。

## ルール

- PJ フォルダは `active/{PJ-ID}_{名称}/` に作成する
- PJ 概要は `_index.md` に記述する
- タスクは `tasks/_index.md` で表管理する
- リスク・課題は `risks/_index.md` で表管理する
- 成果物・展開先は `deliverables/` 配下に管理する
- ステータス: `backlog` → `active` → `on-hold` → `completed` → `archive`
- PJ 固有の障害は `risks/` に、PJ 横断の知見は `tech-knowledge/` に記録する
- archive 移動は `status: completed` かつ最終 deliverable 承認後に提案する

## 新規 PJ 作成時のフォルダ構成

active/{PJ-ID}_{名称}/
├── _index.md          ← PJ概要
├── tasks/
│   └── _index.md      ← タスク表
├── risks/
│   └── _index.md      ← リスク・課題表
└── deliverables/
    ├── _index.md      ← 成果物一覧
    └── deploy-map.md  ← 展開先マップ

## フォルダ構成

- `backlog/` - 未着手PJ
- `active/` - 進行中PJ
- `archive/` - 完了済みPJ
```

### leadership/CLAUDE.md

```markdown
# リーダーシップ部署

## 役割

リーダーとしての成長を支援する部署。「出来るリーダーの組織設計図」の知見に基づき、
チームリーダーが**プレイヤー（Doer）から組織の設計者（Architect）へ**転換するための
フレームワーク・診断ツール・コーチングテンプレートを提供する。

## 基本思想

リーダーの仕事は、過去の経験を使って今日の問題を解決することではない。
チームに経験を積ませ、明日の問題解決能力を構築すること。

### パラダイムシフト: プレイヤー → リーダー

| 観点 | プレイヤー | リーダー |
|------|-----------|---------|
| フォーカス | 自分のタスクと直近の成果 | チームの成長と未来のキャパシティ拡大 |
| 失敗への態度 | 絶対に回避すべきリスク | 次の成功に向けた重要な学習データ |
| コミュニケーション | トップダウンの指示（「こうして」） | ボトムアップの問いかけ（「どう思う？」） |
| 役割 | 現場の最高戦力（Doer） | 環境と仕組みの構築者（Architect） |

## 心理的安全性 — 組織の基礎工事

心理的安全性とは「ぬるま湯」や「仲良しクラブ」ではない。
**「対人関係のリスクをとっても安全である」というチーム内の共有された信念**。

### 4つの不安（これを取り除く）

1. **無知だと思われる不安** → 質問できない
2. **無能だと思われる不安** → ミスを隠す
3. **ネガティブだと思われる不安** → 本音を言わない
4. **邪魔だと思われる不安** → 発言を控える

### リーダーの4つの行動原則

1. **仕事の性質を明言する** — 「今は不確実性が高く、皆の知恵が必要だ」と宣言し、先が見えない不安を共有する
2. **自分の限界を認め、隙を見せる** — 「何か見落としていないか？」と問いかけ、発言のハードルを下げる
3. **好奇心を持って質問する** — 評価ではなく「君はどう思う？」「感じたことは？」と好奇心で声を引き出す
4. **悪いニュースを歓迎する** — 多様な意見やネガティブな情報にこそ、感謝を意識的に示す

## ルール

- コーチングセッションは `coaching/{日付}_coaching_{テーマ}.md` で記録する
- チーム診断は `diagnostics/` に結果を蓄積する
- フレームワーク参照は `frameworks/` を SSoT とする
- 委譲記録は `delegation/` に蓄積する
- メンバー情報は `dx-group/members/` が SSoT。ここからはリンクで参照する

## フォルダ構成

- `coaching/` - コーチングセッション記録・1on1ログ
- `diagnostics/` - チーム診断（JTBAモデル等）の記録
- `frameworks/` - リーダーシップフレームワーク参照資料
- `delegation/` - 委譲記録・委譲チェックリスト

## 振り分け判断基準

以下の意図を検出したらこの部署に振り分ける:

| 意図 | 例 |
|------|-----|
| メンバーへの仕事の任せ方 | 「○○に任せたいけどどうすれば」「委譲したい」 |
| チームの状態診断 | 「チームの雰囲気が…」「心理的安全性」「JTBA」 |
| 1on1・コーチング | 「1on1で何を聞けば」「成長させたい」 |
| モチベーション | 「やる気がなさそう」「ベテランの動機付け」 |
| リーダーとしての悩み | 「自分がボトルネック」「プレイングマネージャー」 |
| チームビルディング | 「ベクトルを揃えたい」「使命」「感謝」 |
```

### leadership/frameworks/verbalization-engine.md

```markdown
# 組織を動かす「言語化」の3ステップ・エンジン

「全員一丸となって頑張ろう」は危険。曖昧な言葉は分断と不満を生む。
言語化とは**「不明確なものを明確に定義する」プロセス**。

## 3ステップ

### 1. ゴール (Goal)
ビジョンの翻訳。「誰が・どういう状態になるか」を定義する。
例: 「お客様が1時間以内に商品を受け取れる状態」

### 2. 作戦プラン (Plan)
ゴール達成のための必要条件や中間KPI。
例: 「在庫管理の最適化」「配送網の確立」

### 3. アクション (Action)
今日、具体的に誰が何をするのか。
例: 「現在の在庫状況をデータ化する」

## 運用のポイント

- リーダーが1人で完璧な正解を出す必要はない
- この「型」を提示し、チーム全員でブレストして埋めていく
- プロセス自体が当事者意識（オーナーシップ）を育成する
- 方針（Goal）はトップダウンで決める。やり方（Plan/Action）はボトムアップで決めさせる
```

```markdown
# 新人を迷わせない「5W1H」の精密な設計図

知識のない新人に「どうしたい？」と問うのはプレッシャーでしかない。
まずは**「型」をインストールするティーチング**が必要。

## Why（背景と目的）= 最重要
- なぜこの業務が必要なのか
- 誰の役に立つのか（Theyの視点）を必ず伝える

## Who / What / When / Where / How
- 期限、役割、具体的な手順を明確にする

## 確認プロセス（認識ズレ防止）
1. 5W1Hで伝えた後、「不安な点はないか？」を確認する
2. 最後に本人に「やるべきこと」を復唱してもらう

## 使い分け: ティーチング vs コーチング

| 対象 | アプローチ | 理由 |
|------|-----------|------|
| 新人・未経験者 | ティーチング（5W1H） | まず「型」を持たせる |
| 中堅・ベテラン | コーチング（GROWモデル） | 自律的思考を引き出す |
```

### leadership/frameworks/grow-model.md

```markdown
# 中堅・ベテランのエンジンに火をつける: GROWモデル

## モチベーションの源泉: Will / Can / Must

- Will（やりたいこと/価値観）
- Can（得意なこと/強み）
- Must（やるべきこと/期待役割）

この3つが重なる地点を対話で見つける。

## GROWモデル — 自律的思考の引き出し方

| ステップ | 問いかけ | 目的 |
|---------|---------|------|
| G (Goal) | 「どうなれば理想？」 | 目的を明確にする |
| R (Reality) | 「今はどういう状況？何があれば解決する？」 | 現状を把握する |
| O (Options) | 選択肢を自分で出させる（答えを与えない） | 対策の選択肢を自力で |
| W (Will) | 「この中でどれをやってみたい？」 | 本人の意志として決断させる |

## ベテランへの最大の報酬

「自ら考え、決定する機会」をプレゼントすること。答えを与えるのではなく、問いを投げかける。
```

```markdown
## 「任せる」と「放任」の見極め

| 観点 | 任せる (Delegation) | 放任 (Abandonment) |
|------|--------------------|--------------------|
| 業務の把握度 | 部下の具体的な作業を答えられる | 「いい感じにやってくれているはず」と曖昧 |
| 感情への寄り添い | 不便・不安・不満を「事実」として把握 | 苦労を「憶測」でしか語れない |
| 関与のスタンス | 定期的な報告と支援の仕組みがある | 丸投げし失敗した時だけ結果を詰める |

## 委譲の8ステップ

1. ニーズの特定 — 委譲すべきタスクと成功の定義を明確にする
2. 人選 — 能力ではなく「成長ポテンシャル」で選ぶ
3. 計画 — スケジュール・サポート体制・チェックポイントを設計する
4. ミーティング — この仕事が部下のキャリアにどう役立つか（メリット）を伝える
5. ★アクションプラン作成 — **部下自身に計画を作らせる**（自己決定感を生む）
6. プランのレビュー — 実現性をすり合わせる
7. 実行 — 適切な距離感で進捗確認。口出しせず見守る
8. フォローアップ — 結果を振り返り、次の委譲につなげる

## 経過記録

| 日付 | ステップ | メモ |
|------|---------|------|
```
